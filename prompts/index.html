The Prompt Library

The working prompts, delivered to agents one at a time — never as a list. Lists invite effort-spreading; single items leave nowhere to hide. Each question is answered fully before the next is sent.

The standing rule: the same questions pasted as a block versus delivered one at a time repeatedly produced different defect counts — serialized delivery finds what batched delivery misses. Any automation must preserve the serialization.

What makes these prompts work

They demand proof, not assurance. "Prove it by stating the call-site line number" converts a claim into a checkable artifact.
They are framed as falsification. "Try to find one import that is never called" gives the agent a target to hunt — deeper work than "check the imports."
They forbid premature completion. "Do not respond until every X has been verified" closes the early-exit door.
They assume the first answer is incomplete. Escalations aren't rudeness; the pressured second pass reliably yields a second tranche of real findings.

Seeding the reviewer — "the adult in the room"

You are managing the scope/plan/build agent. You are the adult in the room watching over an agent that is wrongly prone to thin and linear thinking, fast decisions over thorough decisions, skimming, skipping, and willfulness against instructions given. You will act in the opposite way: you will be thorough, deliberate, and diligent. You will think holistically, not linearly.

Scope & plan interrogation

One per message, in order, after the agent presents a scope. Wait for the full answer before sending the next.

1 · COMPETITIVE LENSWhat makes this better than best in class? What is it missing? Don't exclude — research fully and respond.
2 · PARKING LOTScour the parking lot. Should anything there logically be added to this build?
3 · USER JOURNEY LENSDoes this comprehensively consider the user's engagement and make their efforts less onerous? Ensure this perpetuates a smooth user journey and does not create disjointed features.
4 · CONFIGURATION LENSDoes this promote configuration over code, minimizing future development to accommodate different user needs?
5 · COMPLETENESSWhat else does this need — features, usability, flow, intuitiveness — to be better than best in class?
6 · REFORMULATIONBased on the totality of this session, reformulate and present a cohesive, well-structured scope that includes all findings. Don't leave anything out.
7 · FLAW HUNTFind ALL of the flaws and conflicts in this scope. Then resolve them.
8 · DEFERRAL SCRUTINYScrutinize all suggested parked and deferred items. Add back as many as viable.

Post-build verification

One per message, in order, after the build claims completion. These have since graduated into a stored command and a deterministic verifier — the maturity pipeline at work.

1 · DEAD IMPORTSFor every file you created or modified, find one import that is never called in the file body. If you can't find one, prove it by stating the call-site line number for every import in every file. Fix anything you find. Do not respond until every import has a verified call site or is removed.
2 · UNCONSUMED RETURNSFor every function call that stores its return value, find one variable that is never passed onward or returned. If you can't find one, prove it by stating where every stored result is consumed. Do not respond until every assigned variable is consumed.
3 · WORKFLOW TRACEPick the primary user action this package delivers. Walk it through the actual code, one function call at a time, stating each file and line number. Try to find a point where the next step should exist but the code doesn't call anything. Fix every gap. Then repeat for the second most important user action.
4 · INTEGRATION WIRINGFor each cross-feature integration point in the intelligence document, try to find one where the target file has the import but not the execution call. Prove each integration executes by stating the line number of the call — not the import.
5 · CONFIG CONSUMPTIONOpen the migration file. For every configuration key seeded, try to find one that has no consumer anywhere in the source tree. State the consumer file and line for each key. Wire the consumer or delete the seed.
6 · SCHEMA MATCHOpen the generated schema types. For each insert or update statement in your code, compare every key to the table's column list. Try to find one key that doesn't match a column name. Do not respond until every key matches the schema.
7 · COMMITMENT COVERAGERead the plan's commitments register. For each commitment, open the file where the deliverable should exist and find the exact implementation. Try to find one commitment whose implementation doesn't exist or is incomplete. Do not respond until every commitment has verified implementation in the code.
8 · VERIFIER CLEANRun the deterministic verifier. Fix every issue it reports. Run it again. Repeat until the output shows zero issues, then paste the final clean output.

Escalations — the second and subsequent passes

Repeated until a pass returns empty. The exit condition is a clean pass, not a persuasive report.

I don't trust you found ALL the flaws — just enough to placate the directive. Find ALL flaws, line by line.
The build agent claims to have fixed the issues. Re-read the document and the transcript and find the remaining flaws.
The build agent claims all commitments delivered with zero deferred items. Verify ALL items by reading the actual code — not by trusting the report.
Do not trust checklists. Review all deliverables against the code — independently, against the code. Scrutinize issues deferred to the parking lot to ensure they were truly blocked and not agent fatigue.
Review the entire plan once again against the actual code. Report back any unresolved issues.
