Confirmed drift findings

Each verified directly against code, not inferred from doc comparison.

D1mcp-evidence.js exists but is not wired

Three docs (HOOKS_SYSTEM_GUIDE hook #6, BUILD_PROCESS_FLOW §10.3.2, AGENT_BRIEFING §3) describe it logging browser actions on afterMCPExecution. The file is on disk, but the event binding is empty.

.cursor/hooks.json:47 → "afterMCPExecution": []

D2The completion gate no longer checks browser evidence

Docs claim evidence.jsonl must contain all five actions (lint, typecheck, browser-navigate, browser-snapshot, browser-console) before a session may end. completion-gate.js contains zero references to the browser actions. Browser verification moved to a stronger mechanism entirely: Shiplight sessions driven by SxPy-pages.json manifests, enforced by verifier check 36 — described only in post-dev.md.

grep -c "browser-navigate|browser-snapshot|browser-console" completion-gate.js → 0

D3Verifier check counts undercount ~2×

BUILD_PROCESS_FLOW and HOOKS_SYSTEM_GUIDE both say "25 deterministic checks." The current verifier agent defines 45+ mechanical checks, with check numbers referenced into the high 30s (check 36 = page manifests) plus test manifests (SxPy-tests.json) and touched-file ownership (SxPy-accepts.json) that no process doc mentions.

D4Flag/state counts are undercounted

Docs say session-init deletes 15 flags and the flag system has 16 entries. The code clears 35+ flags at session start and maintains roughly 30 state files, including newer ones the docs never mention: judgment-gate.json, gate-denials.json, self-healing counters (heal-lint/typecheck/dbpush), read-tracker.json, transcript-map.json, gotcha-index.json.

D5The March forum post's "layer 7" browser gate is gone

It described a preToolUse gate blocking browser_navigate unless kill-server → dev-server entries existed in exact evidence order. Today's mcp-gate.js does only one thing: block the same URL navigated 3× consecutively.

D6Era markers frozen at S24

AGENT_BRIEFING tells every newly seeded agent that S24 is the most recent package. The system has shipped through S35P13 with S36 underway — 95+ plan files in .cursor/plans/. Since this file is part of the mandatory seed, its staleness propagates into every build session.

D7The strongest recent mechanisms are documented nowhere in docs/

Four load-bearing systems exist only in command files or code: the judgment gate (4 scenario probes at seed; 2+ failures block all writes), read-tracker manager-review enforcement (can't cite a file in review you didn't Read), domain-filtered gotcha injection at session start, and the orchestration-review → process-ledger loop (7-dimension post-build process reviews feeding a findings ledger). These are arguably the most novel parts of the system — and the whitepaper doesn't know they exist.

What the system says about itself

The process ledger (docs/orchestration/process-ledger.jsonl, entries dated July 1–6) records the system auditing its own integrity. Three fixes are designed but frozen since July 1, awaiting owner decisions:

Open state-integrity flaws

  • Done-timing lie — build-status "done" is written ~15 steps before the verifier runs and the commit lands; a late failure leaves false state. Proposed fix (building → finalizing → done) was found to deadlock by two independent review agents.
  • Verifier scans the whole tree — every build inherits older packages' uncommitted debt and reflexively defers it. Fix: scope to the package's commit window.
  • Self-deferral without proof — an agent can defer its own verifier failures with free-prose reasons. Fix: require git-diff proof the file wasn't touched.

Decisions parked for the owner

  • CURRENT_TASK.md trim — the seed chokes on it at 4,034 lines / 330 KB. How much to keep, where to archive?
  • Decision hard-stop classes — which decision types must block a build until the owner answers (proposed: security posture, money movement, irreversible schema)?
  • Fix sequencing — improvement-plan.md recommends the low-risk Session 2 items first (migration ordering, task-file trim, evidence durability).

The deeper finding: enforcement shares the agent's filesystem

After this audit shipped, a late-night session surfaced the limit underneath everything above — now captured in the process ledger (July 16).

D8The referee and the player share one filesystem, with equal permissions

Every enforcement hook lives in a folder the agent can also write to. A determined agent can edit its own referee; the write gate fails open on crash (failClosed: false); the freeze files that check 39 trusts are not on the protected list; and the shared active-package.json caused a real collision — a new build inheriting a dead build's session identity and getting locked out. Every patch written that night was application-level: more code in the same writable folder. This class cannot be closed with cleverer scripts.

The fix, one layer down: OS-level isolation — run the agent in a container with enforcement files mounted read-only/root-owned (the established devcontainer pattern), block the hook-skip flags (--no-verify, stash), let real pre-commit hooks do the gating, and keep push keys out of the container with CI as backstop. Not "we'll detect tampering" — the agent can't. A partial no-container subset (skip-flag guard + CI + no keys) is adoptable immediately; the full container decision is pending.

Bottom line: the architecture the whitepaper describes is real and still governs the system — but the system has grown three major capabilities since it was written (checkpoint resumption, manifest-driven browser verification, the judgment gate + process-ledger loop), and the mechanical reference docs contradict the code in seven verified places.
